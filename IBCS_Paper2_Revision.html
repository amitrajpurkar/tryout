Option A  ·  Databases
Core idea: How data is stored, structured, and queried.

A1 — Database Concepts

A1.1

Data vs Information

Gist

Data is raw, unprocessed facts with no context; information is data that has been processed, organised, or given meaning so that it is useful to a decision-maker. The distinction matters in databases because a database stores data, but the queries and reports it produces deliver information.

Key line-items
  • Data — raw facts, figures, or symbols (e.g., 42, "Smith", 2024-03-15)
  • Information — data processed into a meaningful, contextual form (e.g., "Smith's order placed on 15 March 2024 totals £42")
  • Processing steps: collecting → organising → structuring → presenting
  • Quality of information depends on: accuracy, completeness, timeliness, relevance
  • Metadata — data about data (e.g., field names, data types, creation date of a record)
Examples
  • 85 is data; "Amit scored 85% in Paper 2" is information.
  • A temperature sensor outputs 23.4 (data); a weather app shows "Currently 23°C, feels warm" (information).
  • Exam question style: "Distinguish between data and information, using an example from a student database."
A1.2

Flat-file vs Relational Databases

Gist

A flat-file database stores everything in a single table (like a spreadsheet), leading to data redundancy and update anomalies. A relational database splits data into multiple linked tables, eliminating redundancy and making maintenance far more efficient.

Key line-items
  • Flat-file database — single table; all data in one place
    • Advantages: simple, easy to create, no specialist software needed
    • Disadvantages: data redundancy, update anomalies, insertion anomalies, deletion anomalies
  • Relational database — multiple tables linked by keys
    • Advantages: no/minimal redundancy, easier updates, data integrity, supports complex queries
    • Disadvantages: more complex to design, requires DBMS software
  • DBMS — software layer managing storage, access, and security (e.g., MySQL, PostgreSQL)
  • Redundancy — the same piece of data stored in more than one place
  • Update anomaly — changing a value in one row but forgetting others
  • Insertion anomaly — cannot add data without unrelated mandatory fields
  • Deletion anomaly — deleting one row loses unrelated data
Examples
  • Flat-file Orders table: every row repeats the customer's name and address → update anomaly if address changes.
  • Relational fix: Customers + Orders linked by CustomerID — address stored once.
  • Exam question style: "Explain one disadvantage of using a flat-file database to store this data."

A2 — Relational Databases

A2.1

Tables, Records, Fields

Gist

The foundational vocabulary of a relational database. A table holds data about one entity type; each row is one instance; each column is one characteristic.

Key line-items
  • Table (relation) — a 2D structure holding data about one entity; has rows and columns
  • Record (row / tuple) — one instance/entry in a table (e.g., one student)
  • Field (column / attribute) — one characteristic of the entity (e.g., StudentName)
  • Domain — the set of permitted values for a field
  • Degree — number of fields (columns) in a table
  • Cardinality — number of records (rows) in a table
  • Null — a field value that is absent or unknown (not zero or empty string)
Examples
  • Table Students(StudentID, Name, DateOfBirth, Grade) has degree 4.
  • One row (1001, "Ana", "2007-04-10", 11) is one record.
  • Exam question style: "Using the table above, state the number of records and fields."
A2.2

Primary Keys & Foreign Keys

Gist

A primary key uniquely identifies each record in its own table; a foreign key in one table references the primary key of another, creating the link between tables. Without keys, joins are impossible and data integrity cannot be enforced.

Key line-items
  • Primary key (PK) — field (or combination) that uniquely identifies every record; must be unique and NOT NULL
  • Composite primary key — PK made of two or more fields; used in junction/linking tables
  • Foreign key (FK) — field in one table that references the PK of another table
  • Referential integrity — FK value must match an existing PK value or be NULL
  • Candidate key — any field that could serve as PK (unique + not null)
  • Natural key — real-world identifier used as PK (e.g., ISBN)
  • Surrogate key — artificial PK with no real-world meaning (e.g., auto-increment ID)
Examples
  • Students(StudentID [PK], Name, Grade) — StudentID is the PK.
  • Enrolments(StudentID [FK], CourseID [FK], EnrolDate) — composite PK = (StudentID, CourseID).
  • Deleting a student whose StudentID appears in Enrolments would violate referential integrity.
  • Exam question style: "Identify the primary key and foreign key(s) in the schema shown."
A2.3

Relationships: 1–1, 1–Many, Many–Many

Gist

Relationships describe how records in one table associate with records in another. A many-to-many relationship always requires a junction table in a relational database.

Key line-items
  • One-to-one (1:1) — one record in Table A links to exactly one in Table B; rare; e.g., Person ↔ Passport
  • One-to-many (1:N) — one record in A links to many in B; most common; FK goes in the "many" side; e.g., Customer → Orders
  • Many-to-many (M:N) — requires a junction/linking table with two FKs; e.g., Students ↔ Courses via Enrolments
Examples
  • Library: Member (1) → Loans (many); Loans → Book (many-to-one). Members ↔ Books is M:N resolved by Loans.
  • Exam question style: "Describe the relationship between Students and Courses. Explain how this is implemented."

A3 — Database Design

A3.1

Entity-Relationship (ER) Models

Gist

An ER diagram is the blueprint for a relational database. It shows entities, their attributes, and relationships between them, including cardinality. Drawing and interpreting ER diagrams is a core exam skill.

Key line-items
  • Entity — a distinct "thing" to store data about; becomes a table; drawn as a rectangle
  • Attribute — a property of an entity; becomes a field; drawn as an ellipse
  • Relationship — association between entities; drawn as a diamond with labelled line
  • Cardinality annotated on lines: 1, N, M, or crow's-foot notation
  • Steps: identify entities → attributes → relationships → cardinality → resolve M:N
  • PK attribute is typically underlined in the diagram
Examples
  • Hospital: Patient — has-many — Appointment — involves — Doctor; Patient ↔ Doctor is M:N resolved by Appointment.
  • Exam question style: "Draw an ER diagram for a booking system with Customers, Rooms, and Bookings."
A3.2

Normalization (1NF, 2NF, 3NF)

Gist

Normalization systematically restructures a database to reduce redundancy and improve integrity. Each normal form removes a specific type of dependency. The IB expects you to recognise violations and know how to fix them up to 3NF.

Key line-items
  • 1NF — every cell contains one atomic value; no lists in a cell; unique column names; row order irrelevant
  • 2NF — 1NF + no partial dependency (every non-key attribute depends on the whole PK; only relevant with composite PKs)
  • 3NF — 2NF + no transitive dependency (no non-key attribute depends on another non-key attribute)
  • Functional dependency — A → B means "knowing A determines B"
Examples
  • 1NF violation: PhoneNumbers = "07123, 07456" → split into separate rows.
  • 2NF violation: OrderDetails(OrderID, ProductID, ProductName, Qty) — ProductName depends only on ProductID → move to Products table.
  • 3NF violation: Employee(EmpID, DeptID, DeptName) — DeptName depends on DeptID, not EmpID → move to Departments table.
  • Exam question style: "The table below is not in 2NF. Identify the partial dependency and show the tables after normalisation."

A4 — SQL

A4.1

SELECT — Querying Data

Gist

SELECT is the most important SQL command for exams. It retrieves data from one or more tables and can filter, sort, and join results. Mastering clause order and JOIN syntax is essential.

Key line-items
  • Basic syntax: SELECT col FROM table WHERE condition ORDER BY col ASC|DESC;
  • SELECT * — retrieve all columns
  • WHERE operators: =, <>, <, >, LIKE, IN, BETWEEN, AND, OR
  • LIKE wildcards: % = any sequence, _ = exactly one character
  • Aggregate functions: COUNT(), SUM(), AVG(), MAX(), MIN()
  • GROUP BY — group rows for aggregates; HAVING — filter groups
  • INNER JOIN — only matching rows; LEFT JOIN — all left rows + matched right
Examples
-- All students in Grade 11, alphabetically
SELECT Name, Grade
FROM Students
WHERE Grade = 11
ORDER BY Name ASC;

-- Count of orders per customer with more than 5 orders
SELECT CustomerID, COUNT(*) AS OrderCount
FROM Orders
GROUP BY CustomerID
HAVING COUNT(*) > 5;

-- Join: student names with their course names
SELECT s.Name, c.CourseName
FROM Students s
JOIN Enrolments e ON s.StudentID = e.StudentID
JOIN Courses c ON e.CourseID = c.CourseID;
A4.2

INSERT, UPDATE, DELETE

Gist

These three DML commands modify data. Exams often ask you to write or trace them, and to understand the danger of a missing WHERE clause.

Key line-items
  • INSERT: INSERT INTO table (col1, col2) VALUES (val1, val2);
  • UPDATE: UPDATE table SET col1 = val WHERE condition; — without WHERE, every row is updated
  • DELETE: DELETE FROM table WHERE condition; — without WHERE, all rows deleted (structure remains)
  • DROP TABLE removes the table entirely; DELETE only removes rows
Examples
-- Add a new student
INSERT INTO Students (StudentID, Name, Grade)
VALUES (1042, 'Priya', 11);

-- Update a student's grade
UPDATE Students SET Grade = 12 WHERE StudentID = 1042;

-- Remove a student record
DELETE FROM Students WHERE StudentID = 1042;

Exam question style: "Write an SQL statement to update the price of product 'P005' to 29.99."

A5 — Data Integrity & Security

A5.1

Validation & Verification

Gist

Validation checks that data conforms to expected rules before acceptance; verification checks that data has been entered accurately against the original source. Both reduce the risk of incorrect data.

Key line-items
  • Type check — correct data type (e.g., integer, date)
  • Range check — value within an acceptable range (e.g., mark 0–100)
  • Length check — string not too long or too short
  • Presence check — field cannot be empty / NULL
  • Format check — matches a pattern (e.g., email must contain @)
  • Check digit — calculated digit appended to verify code (e.g., ISBN-13)
  • Double entry (verification) — data typed twice; discrepancy flagged
  • Proof-reading (verification) — human reviews output against source document
  • Validation ≠ Verification: validation = wrong format; verification = transcription error
Examples
  • Age = −5 fails a range check; date "31/02/2024" fails a format/calendar check.
  • Exam question style: "Describe two validation checks that could be applied to the 'Date of Birth' field."
A5.2

Access Control

Gist

Access control restricts who can see or modify data. It is the primary tool for maintaining confidentiality and data integrity in a multi-user DBMS.

Key line-items
  • Authentication — verifying identity (username + password, biometrics, 2FA)
  • Authorisation — determining what an authenticated user may do
  • Privileges: SELECT, INSERT, UPDATE, DELETE, CREATE, DROP
  • SQL GRANT SELECT ON Students TO 'teacher_user'; and REVOKE
  • RBAC — Role-Based Access Control; permissions assigned to roles
  • Views — virtual tables restricting which rows/columns a user sees
  • Encryption — data at rest or in transit encrypted
  • Audit logs — record all database actions for accountability
Examples
  • School DBMS: Student role can SELECT own record; Teacher can SELECT all; Admin can INSERT/UPDATE/DELETE.
  • Exam question style: "Explain two methods a database administrator could use to protect student data."
Option B  ·  Modelling & Simulation
Core idea: Using computational models to represent real-world systems.

B1 — Principles of Modelling

B1.1

Abstraction

Gist

Abstraction is the process of simplifying a complex real-world system by including only the details relevant to the model's purpose, and deliberately ignoring everything else. Without abstraction, a model would be as complex as reality itself — and therefore useless.

Key line-items
  • Abstraction — removing unnecessary detail to focus on what matters for the model's purpose
  • A model is always a simplification — no model is perfectly accurate
  • What is omitted depends on what questions the model must answer (purpose-driven)
  • Abstraction enables: faster computation, clearer analysis, generalisability
  • Examples of abstracted elements: ignoring air resistance in a projectile model; treating all cars as identical particles in a traffic model
Examples
  • Weather model: abstracts the atmosphere into grid cells with temperature, pressure, humidity — ignores individual molecules.
  • Exam question style: "Explain how abstraction is used when creating a simulation of road traffic."
B1.2

Assumptions and Limitations

Gist

Every model rests on assumptions — simplifications accepted as true for the model to work. These assumptions are the source of the model's limitations. Identifying and evaluating assumptions is a key exam skill.

Key line-items
  • Assumption — a condition accepted as true to make the model workable
  • Limitation — a way in which model output may differ from reality, often caused by an assumption
  • Common assumptions: constant rates, closed systems, uniform behaviour, linear relationships
  • Sensitivity analysis — testing how much output changes when assumptions are varied
  • More assumptions → simpler model → faster but less accurate → less realistic
Examples
  • Disease model assumes constant R0 — reality varies by individual and environment.
  • Exam question style: "State one assumption made in this simulation and explain how it limits the model's accuracy."

B2 — Types of Simulations

B2.1

Discrete vs Continuous

Gist

Discrete simulations change state at specific events or time steps; continuous simulations change state constantly according to mathematical equations. Choosing the right type depends on the system.

Key line-items
  • Discrete — state changes at distinct time steps or events; easier to implement; e.g., daily case counts, queuing model
  • Continuous — state changes are instantaneous and ongoing; requires numerical integration; e.g., fluid flow, projectile motion
  • Computers always discretise time; step size determines accuracy
Examples
  • COVID-19 daily case counts → discrete; aircraft wing stress → continuous.
  • Exam question style: "Explain why a discrete simulation might be chosen to model customer queuing."
B2.2

Deterministic vs Stochastic

Gist

Deterministic simulations always produce the same output for the same input. Stochastic simulations include randomness, so outputs vary. Real-world systems usually have randomness, making stochastic models more realistic but harder to analyse.

Key line-items
  • Deterministic — same input → always same output; no randomness; e.g., billiard ball trajectory
  • Stochastic — includes random variables; run many times to get distributions; e.g., customer arrival simulation
  • Monte Carlo method — repeated random sampling to estimate outcomes
  • Seed — initial RNG value; same seed → same "random" sequence (enables reproducibility)
Examples
  • Planet orbit = deterministic; queuing simulation with random arrivals = stochastic (Monte Carlo).
  • Exam question style: "State one reason why a stochastic simulation is more appropriate for modelling customer arrivals."

B3 — Developing Models

B3.1

Variables & Parameters

Gist

Variables change as the simulation runs; parameters are fixed values set before the simulation starts. Understanding this distinction is essential for designing and critiquing a model.

Key line-items
  • Variable — quantity whose value changes during simulation (e.g., population size, temperature)
  • Parameter — fixed input set before the run (e.g., birth rate, infection probability, speed limit)
  • Initial conditions — starting values of all state variables at time = 0
  • Changing parameters = "what-if" scenario analysis
  • Time step (Δt) — increment between updates; smaller Δt → more accurate but slower
Examples
  • Population model: variable = current population N; parameters = birth rate r, carrying capacity K.
  • Exam question style: "Identify one variable and one parameter in the simulation described."
B3.2

Algorithms for Simulation

Gist

The simulation algorithm updates the system's state at each time step. Tracing or writing pseudocode for simple simulation loops is an exam skill.

Key line-items
  • Loop structure: Initialise → Repeat (calculate new state → record → advance time) → Analyse
  • Euler method: new_value = old_value + rate × Δt
  • Event-driven simulation — advances to next event; efficient for sparse events
  • Validation — comparing output to real-world data (is the model correct?)
  • Verification — checking code implements intended algorithm (is the code correct?)
Examples
  • Population growth (Euler): N_new = N + (birth_rate - death_rate) × N × Δt
  • Exam question style: "Trace the algorithm for the first three time steps of this population simulation."

B4 — Applications

B4.1

Scientific Simulations

Gist

Simulations are used across science when real experiments are impossible, too dangerous, too expensive, or too slow. Knowing common application areas and why simulation is preferred is an important exam point.

Key line-items
  • Why simulate? Too dangerous / expensive / slow / ethically prohibited / impossible to observe directly
  • Weather modelling — 3D grid cells, fluid dynamics equations, updated every few minutes
  • Traffic simulation — optimises signal timing, detects congestion
  • SIR epidemic model — Susceptible → Infected → Recovered; evaluates interventions
  • Population models — predator-prey, resource consumption
  • Structural engineering — finite element analysis of bridges under load
Examples
  • SIR model helped governments decide lockdown timing during COVID-19.
  • Exam question style: "Describe two advantages of simulation over a physical experiment for vehicle crash safety."

B5 — Evaluation

B5.1

Accuracy vs Realism

Gist

A simulation can be accurate without being fully realistic, and vice versa. The trade-off between accuracy and computational cost is central to evaluating any model.

Key line-items
  • Accuracy — how closely output matches observed real-world data
  • Realism — how closely internal mechanics reflect actual processes
  • Calibration — adjusting parameters to match known historical data
  • Validation — testing calibrated model against new, unseen data
  • More complex ≠ more accurate; over-fitting and incorrect assumptions can reduce accuracy
Examples
  • Simple SIR model may match overall epidemic curve (accurate) but ignores age stratification (not fully realistic).
  • Exam question style: "Evaluate the accuracy of this simulation by identifying two assumptions that may reduce its reliability."
B5.2

Ethical Implications

Gist

Simulations influence major decisions in healthcare, policy, finance, and engineering. When models are wrong or misused, the consequences can be severe. Ethical awareness is expected by the IB.

Key line-items
  • Over-reliance on model output without understanding limitations
  • Assumptions can embed bias (demographic assumptions in health models)
  • Results can be selectively reported to support a pre-determined conclusion
  • Privacy: personal data used to train/validate a model
  • Transparency and reproducibility required for public-policy models
  • Environmental impact of large-scale simulation (energy consumption)
Examples
  • Financial risk models failed to predict the 2008 crash → over-confidence without stress-testing edge cases.
  • Exam question style: "Discuss one ethical issue arising from using a simulation to make public health decisions."
Option C  ·  Web Science
Core idea: How the web works and its societal impact.

C1 — Internet Fundamentals

C1.1

TCP/IP and DNS

Gist

The internet runs on TCP/IP — protocols that break data into packets, route them across networks, and reassemble them at the destination. DNS translates human-readable domain names into the IP addresses computers use to locate each other.

Key line-items
  • Protocol — agreed rules for communication between devices
  • IP — addressing and routing of packets; every device has a unique IP address
  • TCP — reliable, ordered delivery; breaks data into packets; acknowledges receipt; retransmits lost packets
  • Packet — small chunk of data: header (source IP, dest IP, sequence no.) + payload
  • Packet switching — packets may travel different routes; robust, no single point of failure
  • IPv4: 32-bit (e.g., 192.168.1.1); IPv6: 128-bit (e.g., 2001:0db8::1)
  • DNS — translates domain names to IP addresses; hierarchy: root → TLD → authoritative server
  • Port — identifies a process/service: HTTP = 80, HTTPS = 443, FTP = 21
Examples
  • Typing www.ibo.org → DNS lookup → IP returned → TCP on port 443 → data transferred in packets.
  • Exam question style: "Outline the role of DNS in accessing a web page."
C1.2

Client-Server Model

Gist

Clients request resources; servers respond. The HTTP request-response cycle, methods, and status codes are the language of the web and regularly appear in exam questions.

Key line-items
  • Client — requests service (e.g., browser); Server — provides service (e.g., web server)
  • HTTP/HTTPS — application-layer protocol for web communication; HTTPS adds TLS encryption
  • HTTP methods: GET (fetch data), POST (send data), PUT (update), DELETE (remove)
  • Status codes: 200 OK, 301 Redirect, 404 Not Found, 500 Internal Server Error
  • Request-response cycle: client sends request → server processes → server returns response
Examples
  • Browser sends GET /index.html HTTP/1.1 → server responds 200 OK + HTML content.
  • Login form data sent via POST (not GET) to hide credentials from the URL.
  • Exam question style: "Describe the client-server interaction when a user submits a search query."

C2 — Web Technologies

C2.1

HTML and CSS Basics

Gist

HTML provides structure and content; CSS controls visual presentation. Together they form the foundation of every webpage. The IB expects basic literacy: tag identification, attribute use, and the CSS box model.

Key line-items
  • HTML uses tags: <h1>, <p>, <a href="">, <img src="" alt="">, <table>, <form>
  • HTML is not a programming language — no logic or loops
  • CSS rule format: selector { property: value; }
  • Selectors: element (p), class (.menu), ID (#header)
  • Box model — content + padding + border + margin
  • CSS can be inline, internal (<style>), or external (linked .css file)
  • Separation of concerns — HTML structure, CSS style, JavaScript behaviour
Examples
<a href="https://www.ibo.org" target="_blank">Visit IBO</a>
<img src="logo.png" alt="School logo" width="200">

h1 { font-family: Arial; color: #333; font-size: 24pt; }

Exam question style: "Write HTML to create a hyperlink that opens in a new tab."

C2.2

Dynamic vs Static Web Pages

Gist

Static pages deliver the same content to every user; dynamic pages are generated on the fly, tailoring content to the user or data. Understanding this distinction explains why most modern websites use scripting.

Key line-items
  • Static — fixed HTML file; same for all visitors; fast, simple, no server processing
  • Dynamic — generated at request time; varies by user, session, or database query
  • Server-side — server script (PHP, Python, Node.js) queries DB, builds HTML, sends to client
  • Client-side — JavaScript runs in browser, updates DOM after page load (AJAX, React)
  • Cookies — small data files stored on client; persist session info (login state, cart)
Examples
  • School homepage with opening hours = static; personalised newsfeed = dynamic.
  • Exam question style: "Explain the difference between a static and dynamic web page in the context of an online shop."

C3 — Search Engines

C3.1

Indexing and Ranking Algorithms

Gist

Search engines work in three stages: crawling, indexing, ranking. The ranking algorithm — notably PageRank — is the commercially critical part.

Key line-items
  • Web crawler (spider/bot) — follows hyperlinks to discover and download pages
  • Inverted index — maps every word to the documents containing it; enables fast keyword lookups
  • PageRank — a page's importance is based on the number and quality of inbound links
  • Ranking factors: keyword relevance, PageRank/authority, freshness, user engagement, personalisation
  • SEO — white-hat (quality content, meta tags, backlinks) vs black-hat (keyword stuffing, link farms)
  • Meta tags — HTML tags providing info to search engines: <meta name="description" content="...">
Examples
  • Searching "IB CS revision" → crawler indexed relevant pages → inverted index finds matching pages → PageRank + relevance determines order.
  • Exam question style: "Explain how a search engine uses a crawler and index to respond to a query."

C4 — Social & Ethical Issues

C4.1

Privacy and Security

Gist

The web collects enormous personal data, creating significant privacy risks. Security threats — from phishing to SQL injection — are constant. The IB expects awareness of risks and countermeasures.

Key line-items
  • Privacy issues: data harvesting by cookies/trackers, data breaches, GDPR requirements
  • Phishing — fake emails/sites tricking users into revealing credentials
  • Man-in-the-middle — attacker intercepts communication
  • SQL injection — malicious SQL inserted via form field to manipulate database
  • XSS — injecting malicious scripts into pages viewed by other users
  • DDoS — flooding a server with requests to make it unavailable
  • Countermeasures: HTTPS/TLS, firewalls, input validation, parameterised queries, 2FA
Examples
  • SQL injection: entering ' OR '1'='1 in a login field could bypass authentication if queries are not parameterised.
  • Exam question style: "Describe two security measures a website could use to protect user data."
C4.2

Digital Divide

Gist

The digital divide is the gap between those with reliable access to digital technology and those without. It has significant social, economic, and educational consequences.

Key line-items
  • Dimensions: global (developed vs developing), social (urban vs rural, wealthy vs poor), gender
  • Causes: cost of devices/connectivity, lack of infrastructure, language barriers, digital literacy
  • Consequences: educational disadvantage, limited economic opportunity, political exclusion
  • Bridging: government investment, low-cost devices, community Wi-Fi, digital literacy programmes
  • Second-level divide — even among those with access, quality of use varies
Examples
  • Rural communities in developing countries without fibre broadband: students lack online learning access.
  • Exam question style: "Discuss the impact of the digital divide on educational opportunities."

C5 — Web Trends

C5.1

Social Media and Big Data

Gist

Social media platforms generate vast quantities of user-generated data, contributing to the "big data" phenomenon. Understanding big data's characteristics and its social/ethical implications is essential.

Key line-items
  • Big data — characterised by the 3 Vs: Volume, Velocity, Variety
  • Data mining — discovering patterns in large datasets
  • Filter bubble — algorithmic curation reinforcing existing views
  • Personalisation — using data to tailor content to individual users
  • Monetisation: user data sold to advertisers — "if the product is free, you are the product"
  • Data sovereignty — questions about who owns and controls personal data
Examples
  • Facebook analyses likes/shares to target advertising; Netflix uses viewing data + ML for personalised recommendations.
  • Exam question style: "Explain two ethical concerns raised by the use of big data by social media companies."
Option D  ·  Object-Oriented Programming
Core idea: Designing software using classes and objects.

D1 — Basic OOP Concepts

D1.1

Classes and Objects

Gist

A class is a blueprint defining a category of things — its characteristics (attributes) and behaviours (methods). An object is a specific instance created from that blueprint. OOP organises code around these real-world abstractions.

Key line-items
  • Class — a user-defined type; defines structure and behaviour shared by all objects of that type
  • Object (instance) — a concrete realisation of a class in memory
  • Constructor — special method called when object is created; initialises attributes
  • this / self — keyword referring to the current object instance
  • State — current values of an object's attributes; Behaviour — its methods
  • Instantiation — the act of creating an object from a class
Examples
public class Dog {
    String name;
    String breed;
    public Dog(String name, String breed) {
        this.name = name;
        this.breed = breed;
    }
    public void bark() {
        System.out.println(name + " says: Woof!");
    }
}
Dog d1 = new Dog("Rex", "Labrador");
d1.bark(); // Rex says: Woof!

Exam question style: "Define 'class' and 'object', giving one example of each from a library system."

D1.2

Attributes and Methods

Gist

Attributes store what an object knows (state); methods define what an object can do (behaviour). Identifying and writing both in code is a foundational OOP exam skill.

Key line-items
  • Instance attribute — unique to each object; created in the constructor
  • Class attribute (static field) — shared by all objects; declared with static in Java
  • void method — returns nothing; performs an action
  • Value-returning method — returns a value of a specified type
  • Method signature — method name + parameter list
  • Calling a method: object.methodName(arguments)
Examples
public class BankAccount {
    private double balance;
    public void deposit(double amount) { balance += amount; }
    public double getBalance() { return balance; }
}

Exam question style: "Identify two attributes and one method for a Product class in an e-commerce system."

D2 — Encapsulation

D2.1

Access Modifiers

Gist

Encapsulation bundles data and methods into a class and restricts direct external access. Access modifiers enforce information hiding, protecting internal state from unintended modification.

Key line-items
  • private — accessible only within the same class (recommended for attributes)
  • public — accessible from anywhere
  • protected — accessible within the class and its subclasses
  • Benefits: protects data integrity, reduces coupling, allows internal changes without breaking external code
Examples
  • private double balance; in BankAccount prevents other classes from setting balance = -999; directly.
  • Exam question style: "Explain why it is good practice to declare attributes as private."
D2.2

Getters and Setters

Gist

Getters and setters provide controlled, indirect access to private attributes. They allow validation logic to be inserted, maintaining data integrity while preserving the interface.

Key line-items
  • Getter (accessor) — returns the value of a private attribute; named getAttributeName()
  • Setter (mutator) — sets the value with validation; named setAttributeName(value)
  • Enables: input validation, read-only attributes (getter without setter), internal representation changes
Examples
public class Student {
    private int age;
    public int getAge() { return age; }
    public void setAge(int age) {
        if (age > 0 && age < 120) { this.age = age; }
    }
}

Exam question style: "Write a getter and setter for the price attribute, ensuring price cannot be negative."

D3 — Inheritance

D3.1

Superclass and Subclass

Gist

Inheritance allows a subclass to acquire the attributes and methods of a superclass, extending or specialising its behaviour. It implements the "is-a" relationship and is the primary mechanism for code reuse in OOP.

Key line-items
  • Superclass (parent) — the class being extended; Subclass (child) — the class that inherits
  • Java syntax: class Dog extends Animal { ... }
  • Subclass inherits all public and protected members
  • Subclass can add new members or override superclass methods
  • super — calls the superclass constructor or method from the subclass
  • Java: single inheritance only; interfaces provide a workaround for multiple inheritance
Examples
class Animal {
    String name;
    public void eat() { System.out.println(name + " eats."); }
}
class Dog extends Animal {
    public void bark() { System.out.println(name + " barks."); }
}
Dog d = new Dog();
d.name = "Rex";
d.eat();  // inherited from Animal
d.bark(); // defined in Dog

Exam question style: "Draw a class hierarchy for a vehicle system. Explain what ElectricCar inherits."

D3.2

Code Reuse

Gist

Inheritance's primary practical benefit is code reuse — writing a method once in a superclass and having it available in all subclasses. This reduces duplication and centralises bug fixes.

Key line-items
  • DRY principle — Don't Repeat Yourself; inheritance achieves this
  • "is-a" test — use inheritance if SubclassObject is-a SuperclassObject (e.g., Dog is-a Animal ✓)
  • "has-a" test — use composition instead (e.g., Car has-a Engine; don't inherit from Engine)
  • Fix in superclass automatically updates all subclasses
  • Inheritance can be overused — deep hierarchies become difficult to maintain
Examples
  • BankAccount has deposit() and withdraw(); SavingsAccount and CurrentAccount inherit these and add their own methods.
  • Exam question style: "Explain how inheritance promotes code reuse in the class hierarchy shown."

D4 — Polymorphism

D4.1

Method Overriding

Gist

Overriding lets a subclass provide its own implementation of a superclass method. At runtime, the correct version is called based on the actual object type — this is dynamic (runtime) polymorphism.

Key line-items
  • Same name, return type, and parameters as the superclass method; different body in the subclass
  • @Override annotation — optional but best practice; compiler verifies the override
  • Dynamic dispatch — at runtime, Java calls the most derived version based on actual object type
  • Superclass reference can point to subclass object: Animal a = new Dog();
  • Abstract method — declared with no body; forces all subclasses to override
  • Abstract class — cannot be instantiated; exists only to be subclassed
Examples
class Animal { public void makeSound() { System.out.println("..."); } }
class Dog extends Animal {
    @Override
    public void makeSound() { System.out.println("Woof!"); }
}
class Cat extends Animal {
    @Override
    public void makeSound() { System.out.println("Meow!"); }
}
Animal a = new Dog();
a.makeSound(); // prints "Woof!" — runtime polymorphism

Exam question style: "Explain how polymorphism is demonstrated above. State the output of each method call."

D4.2

Method Overloading

Gist

Overloading allows multiple methods with the same name in the same class, as long as they have different parameter lists. This is compile-time (static) polymorphism.

Key line-items
  • Same method name, different parameter signature (number, type, or order of parameters)
  • Return type alone is NOT sufficient to distinguish overloads
  • Resolved at compile time (unlike overriding, which is runtime)
  • Constructor overloading — multiple constructors with different parameter lists
  • Key distinction: Overloading = same class, different signatures; Overriding = subclass, same signature
Examples
public class Calculator {
    public int add(int a, int b) { return a + b; }
    public double add(double a, double b) { return a + b; }
    public int add(int a, int b, int c) { return a + b + c; }
}

Exam question style: "Distinguish between method overloading and method overriding."

D5 — Program Design

D5.1

UML Diagrams

Gist

UML class diagrams are the standard notation for visualising OOP designs. They show classes, attributes, methods, and relationships. Reading and drawing them is a testable IB skill.

Key line-items
  • Class box: name (top) | attributes (middle) | methods (bottom)
  • Visibility: + public, - private, # protected
  • Attribute format: - name : DataType
  • Method format: + methodName(param: Type) : ReturnType
  • Inheritance: solid line with open arrowhead pointing to superclass
  • Aggregation: open diamond; Composition: filled diamond; Multiplicity: 1, 0..*, 1..*
Examples
+------------------+
|     Student      |
+------------------+
| - studentID: int |
| - name: String   |
+------------------+
| + getID(): int   |
| + getName(): Str |
+------------------+

Exam question style: "Draw a UML diagram for a Shape superclass and Circle, Rectangle subclasses with two attributes and two methods each."

D5.2

Class Relationships

Gist

Beyond inheritance, classes relate through association, aggregation, and composition. Choosing the right relationship type reflects real-world connections and has practical implications for object lifecycle.

Key line-items
  • Association — "uses" relationship; both can exist independently (e.g., Teacher teaches Course)
  • Aggregation — weak "has-a"; contained object can exist independently (e.g., Department has Employees)
  • Composition — strong "has-a"; part cannot exist without whole (e.g., House has Rooms)
  • Inheritance — "is-a" relationship
  • Dependency — one class temporarily uses another (e.g., method receives object as parameter)
Examples
  • Library aggregates Book — books exist independently of the library.
  • Order composes OrderLine — an order line makes no sense outside an order.
  • Exam question style: "Explain the difference between aggregation and composition with a suitable example."

D6 — Programming Practice

D6.1

Writing and Debugging Code

Gist

Option D is the most code-heavy. Exams include writing, tracing, and debugging Java programs. Knowing error types, debugging strategies, and testing techniques is essential for full marks.

Key line-items
  • Syntax error — violates grammar rules; detected by compiler (e.g., missing semicolon)
  • Logic error — code compiles and runs but produces wrong output; found by testing and tracing
  • Runtime error (exception) — occurs during execution (e.g., NullPointerException, division by zero)
  • Debugging: print statements, rubber duck debugging, IDE debugger with breakpoints
  • White-box testing — tester has source code; tests internal paths
  • Black-box testing — tester knows only inputs/outputs; tests against requirements
  • Boundary testing — testing at the limits of valid input ranges
  • Exception handling: try { } catch (ExType e) { } finally { }
Examples
  • Boundary test for setAge(): test ages −1 (invalid), 0 (boundary), 1 (valid), 119 (valid), 120 (boundary), 121 (invalid).
  • Exam question style: "Identify the error in the following code and explain how it would manifest at runtime."
D6.2

Tracing Algorithms

Gist

Tracing (dry-running) means manually executing code step-by-step, recording variable state in a trace table, and determining output. Exams frequently include trace tables.

Key line-items
  • Trace table — one column per variable plus output; one row per step or iteration
  • Read line by line; update table for each assignment, condition, output statement
  • Common patterns: linear search, bubble sort, recursion (track call stack), accumulator
  • Always note final output and return values
Examples

Trace factorial(3):

factorial(3) → 3 × factorial(2)
  factorial(2) → 2 × factorial(1)
    factorial(1) → 1
  factorial(2) = 2
factorial(3) = 6

Trace table for sum = 0; for i in 1..3: sum += i:

isum
—0
11
23
36

Exam question style: "Trace the algorithm for the input array [5, 3, 8, 1] and complete the trace table."
